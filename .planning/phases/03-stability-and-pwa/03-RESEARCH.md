## Summary

Phase 3 is a hardening phase with no new features — it fixes known bugs, adds resilience to the app shell, and ships a proper PWA manifest for iPhone home screen installation. All seven requirements are well-scoped surgical changes to existing files. The codebase is fully read and the exact lines requiring change are identified below.

The two most consequential changes are the ErrorBoundary (STABLE-01) and the PWA manifest (PWA-01/02). ErrorBoundary requires a `"use client"` class-component wrapper placed around each tab's render site in `app/page.tsx`. The manifest requires a new `app/manifest.ts` file (Next.js App Router convention) plus icon PNGs in `public/`; iOS Safari ignores the manifest for splash screens and instead relies on `apple-touch-startup-image` links, which are already partially handled by the `appleWebApp` metadata in `layout.tsx`.

The remaining four items (STABLE-02 through STABLE-04, PWA-03) are small, contained fixes: remove one `slice(-30)` call, replace `Math.random()` with `crypto.randomUUID()` in three locations, add NaN guards in `TradeForm.tsx`, and confirm `strategyText` is already reactive state (it is — PWA-03 is already done).

**Primary recommendation:** Treat this as five focused tasks: (1) ErrorBoundary component + placement, (2) manifest.ts + icons, (3) UUID migration, (4) NaN input guard, (5) history cap fix + PWA-03 verification. Each is independently testable.

### Pattern 1: TabErrorBoundary (STABLE-01)

**What:** Class component ErrorBoundary marked `"use client"`, wrapping individual tab renders in `page.tsx`

**When to use:** Per-tab isolation — a crash in one tab must not unmount the navigation shell or other tabs

**Important constraint:** In Next.js App Router, `app/page.tsx` is already `"use client"` (it uses hooks). The ErrorBoundary component file must also be `"use client"` since class components with lifecycle methods cannot run on the server.

**React 19 note:** Error boundaries remain class components in React 19. React does not yet support function component error boundaries (this is a known open proposal). The pattern is unchanged from React 16+. [CITED: react.dev/reference/react/Component — getDerivedStateFromError, componentDidCatch]

```typescript
// Source: https://react.dev/reference/react/Component
// app/components/TabErrorBoundary.tsx
"use client"

import { Component, type ReactNode } from "react"

interface Props {
  children: ReactNode
  tabName: string
}

interface State {
  hasError: boolean
  error: Error | null
}

export class TabErrorBoundary extends Component<Props, State> {
  constructor(props: Props) {
    super(props)
    this.state = { hasError: false, error: null }
  }

  static getDerivedStateFromError(error: Error): State {
    return { hasError: true, error }
  }

  componentDidCatch(error: Error, info: { componentStack: string }) {
    console.error(`[TabErrorBoundary:${this.props.tabName}]`, error, info.componentStack)
  }

  render() {
    if (this.state.hasError) {
      return <TabErrorFallback tabName={this.props.tabName} onRetry={() => this.setState({ hasError: false, error: null })} />
    }
    return this.props.children
  }
}
```

**Placement in page.tsx:** Wrap each tab render site individually. The tab navigation buttons, FAB button, TradeModal, and ambient orbs all remain OUTSIDE the ErrorBoundary — they cannot be crashed by a tab component failure.

```tsx
// In page.tsx — wrap each tab at its render site
{showAccount ? (
  <TabErrorBoundary tabName="account"><AccountTab /></TabErrorBoundary>
) : activeTab === "log" ? (
  <>
    {/* toggle buttons — outside boundary */}
    {journalView === "log"
      ? <TabErrorBoundary tabName="log"><LogTab /></TabErrorBoundary>
      : <TabErrorBoundary tabName="stats"><StatsTab /></TabErrorBoundary>}
  </>
) : (
  <TabErrorBoundary tabName={activeTab}><ActiveComponent /></TabErrorBoundary>
)}
// Coach tab — separate flex sibling:
<TabErrorBoundary tabName="coach"><CoachTab /></TabErrorBoundary>
```

**TabErrorFallback design:** Must match Midnight Market. Use `--bg2` background, `--border` rim, `--text2` for message text, `--accent` for retry button. No emojis per project rules. IBM Plex Mono for any displayed error codes.

### Pattern 2: Next.js manifest.ts (PWA-01/02)

**What:** File convention — `app/manifest.ts` exports a default function returning `MetadataRoute.Manifest`. Next.js auto-serves it at `/manifest.webmanifest` and generates `<link rel="manifest">` in the HTML head. No manual `<link>` tag needed in `layout.tsx`.

[CITED: nextjs.org/docs/app/api-reference/file-conventions/metadata/manifest]

```typescript
// Source: https://nextjs.org/docs/app/api-reference/file-conventions/metadata/manifest
// app/manifest.ts
import type { MetadataRoute } from "next"

export default function manifest(): MetadataRoute.Manifest {
  return {
    name: "Trading Edge",
    short_name: "Trading Edge",
    description: "Prop futures trading journal & AI coaching",
    start_url: "/",
    display: "standalone",
    background_color: "#060b14",
    theme_color: "#060b14",
    icons: [
      { src: "/icon-192x192.png", sizes: "192x192", type: "image/png" },
      { src: "/icon-512x512.png", sizes: "512x512", type: "image/png", purpose: "maskable" },
    ],
  }
}
```

**PWA-02 note:** `<link rel="manifest">` is automatically injected by Next.js when `app/manifest.ts` exists. No additional change to `layout.tsx` is required for the link itself.

**iOS-specific additions to layout.tsx metadata:** iOS Safari does NOT use the Web App Manifest for the home screen icon or splash screen. It uses Apple-proprietary meta tags. These must be added to the `metadata` export in `layout.tsx`:

- `appleWebApp.capable: true` — already set
- `appleWebApp.statusBarStyle: "black-translucent"` — already set
- `appleWebApp.title` — already set
- `icons.apple` — maps to `<link rel="apple-touch-icon">` — **NOT currently set**

```typescript
// Source: https://nextjs.org/docs/app/api-reference/functions/generate-metadata#applewebapp

  title: "TRADING EDGE",
  description: "Prop futures trading journal & AI coaching",

For the iPhone 17 Pro (393×852pt @3x = 1179×2556px), this is a significant asset generation burden. **Recommendation:** Skip splash screen images in Phase 3. The `background_color: "#060b14"` in the manifest and `statusBarStyle: "black-translucent"` provide a near-seamless dark launch experience without custom splash assets. Splash screen images can be added in Phase 5 Polish.

### Pattern 3: UUID Migration (STABLE-03)

**What:** Replace `Math.random().toString(36)` with `crypto.randomUUID()` in three files.

**`crypto.randomUUID()` availability:** Available in Node.js 19+ and all modern browsers (Chrome 92+, Safari 15.4+, Firefox 95+). The project targets iPhone 17 Pro — Safari 15.4+ is a safe baseline. In Next.js 16 server-side routes, `crypto` is the Node.js global crypto module which has `randomUUID()`. [VERIFIED: `node -e "console.log(crypto.randomUUID())"` succeeds on this machine's Node 24]

**Three locations to change:**

1. `app/components/TradeForm.tsx` line 7 — `genId()` function (used for trade IDs at line 65)
2. `app/components/tabs/Coach.tsx` line 45 — local `genId()` function (used for coaching entry IDs and chat message IDs at lines 452, 521, 590, 626)
3. `app/components/tabs/Stats.tsx` line 281 — fallback `Math.random().toString(36)` (guard for missing trade ID — should also use `crypto.randomUUID()`)

```typescript
// Before:
export function genId() { return Math.random().toString(36).slice(2, 9) }

// After:
export function genId() { return crypto.randomUUID() }
```

**Session ID in Coach.tsx line 52** uses `Math.random()` for `sessionStorage.setItem("edge_session_id", sid)`. This is the server-side rate-limiting key. It does NOT need to be a UUID — it just needs to be unique per browser session. However, for consistency, it should also use `crypto.randomUUID()`.

### Pattern 4: NaN Input Validation (STABLE-04)

**What:** Guard the `submit` handler in `TradeForm.tsx` to reject NaN values before calling `onSubmit`.

**Root cause:** Inputs use `type="text"` with `inputMode="decimal"`. If a user types non-numeric content, `Number("abc")` returns `NaN`. The form currently passes this NaN directly to `addTrade` → stored in Supabase.

**`Number("")` returns `0`** — empty pnl/rmult fields already resolve safely. The only failure case is alphabetical or mixed characters.

**Validation approach:** Check `isNaN` on the parsed numbers. Show an inline error state on the affected field. Do NOT block silently — show the user what's wrong.

```typescript
// In TradeForm.tsx submit handler:
const submit = (e: React.FormEvent) => {
  e.preventDefault()
  const pnlNum = Number(form.pnl)
  const rmultNum = Number(form.rmult)
  if (isNaN(pnlNum)) { setPnlError("Enter a valid number"); return }
  if (isNaN(rmultNum)) { setRmultError("Enter a valid number"); return }
  onSubmit({
    id: crypto.randomUUID(),
    ...
    pnl: pnlNum,
    rmult: rmultNum,
    ...
  })
}
```

**Error display:** Inline text below the field in `var(--red)`, `mono text-xs`. Clear error when field changes.

### Pattern 5: History Cap Fix (STABLE-02)

**Root cause:** Line 191 in `TradesContext.tsx`:
```typescript
// Current (bug):
if (hydrated) localStorage.setItem(COACHING_KEY, JSON.stringify(coachingHistory.slice(-30)))
// Fixed:
if (hydrated) localStorage.setItem(COACHING_KEY, JSON.stringify(coachingHistory.slice(-60)))
```

**In-memory cap is already correct:** Line 271 already does `setCoachingHistory((prev) => [...prev, e].slice(-60))`. The bug is only in the localStorage write — entries 31–60 exist in React state but are truncated when written to the cache. On next reload, the cache hydrates with only 30 entries, then Supabase overwrites with the full 60. The practical effect: between app start and Supabase auth resolving, history shows 30 entries. Fix is a single character change.

### Anti-Patterns to Avoid

- **Wrapping the entire `<main>` in one ErrorBoundary:** This would make a crash in any tab remove ALL tab content, breaking the isolation requirement. Wrap each tab individually.
- **Wrapping the nav shell in ErrorBoundary:** The navigation buttons, FAB, and TradeModal must remain outside all error boundaries — they must survive any tab crash.
- **Using `error.tsx` App Router convention for tab isolation:** App Router's `error.tsx` file convention is for route-segment errors (server components, data fetching). It does not apply here — the entire app is a single route (`app/page.tsx`). We need a manual class-component ErrorBoundary wrapping specific render sites.
- **Placing `manifest.ts` in `public/`:** It must go in `app/manifest.ts` to use the App Router file convention. A static `public/manifest.json` works but loses TypeScript type-checking via `MetadataRoute.Manifest`.
- **Expecting iOS Safari to use manifest `icons` for home screen icon:** iOS uses `apple-touch-icon` from the `<link>` tag, not from the manifest. Both must be provided.
- **Using `display: "fullscreen"` in manifest:** Use `"standalone"` for iOS PWA. `fullscreen` is not well-supported on iOS Safari and removes the status bar area unexpectedly.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| UUID generation | Custom ID scheme | `crypto.randomUUID()` | Built into the platform; collision-resistant UUID v4 |
| Error catching | try/catch wrappers in every tab | Class component ErrorBoundary | React's built-in mechanism for catching render errors; try/catch cannot catch errors thrown during render |

**Key insight:** `crypto.randomUUID()` is available in both browser and Node.js environments used by this project. No package needed.

## Common Pitfalls

### Pitfall 1: ErrorBoundary Must Be a Class Component

**What goes wrong:** Developer tries to write a function component ErrorBoundary using try/catch in a render function or a useEffect — it does not catch render errors.

**Why it happens:** React's error boundary mechanism requires `getDerivedStateFromError` and `componentDidCatch` which only exist on class components.

**How to avoid:** Write the ErrorBoundary as a class component. Mark the file `"use client"`. The class component itself does not need hooks — it just needs React imported.

**Warning signs:** TypeScript errors like "Type 'X' does not satisfy constraint 'Component<P, S>'" or errors not being caught during testing.

### Pitfall 2: ErrorBoundary Doesn't Catch Async Errors or Event Handler Errors

**What goes wrong:** Errors thrown in `useEffect`, `setTimeout`, Promise rejections, or event handlers are NOT caught by ErrorBoundary — only synchronous render errors are caught.

**Why it happens:** React's boundary mechanism hooks into the reconciler, not the event loop.

**How to avoid:** The Coach tab's API call errors are already handled (try/catch in POST handler). Tab render errors (bad state shape, undefined access during render) are the target. This is sufficient for STABLE-01's requirement.

### Pitfall 3: iOS Safari PWA — `<link rel="manifest">` Not Enough for Home Screen Icon

**What goes wrong:** Manifest correctly specifies icons, but the iOS home screen shows a screenshot/default icon instead of the custom icon.

**Why it happens:** iOS Safari ignores the manifest `icons` array for home screen icons. It uses the `apple-touch-icon` link tag exclusively.

**How to avoid:** Add `icons.apple` to the `metadata` export in `layout.tsx` pointing to a 180×180px PNG in `public/`. This generates `<link rel="apple-touch-icon" sizes="180x180" href="/apple-icon-180x180.png">`.

**Warning signs:** Manifest present and valid, but icon shows as webpage screenshot on iOS home screen.

### Pitfall 4: `crypto.randomUUID()` Not Available in Older Safari

**What goes wrong:** Code throws `TypeError: crypto.randomUUID is not a function` on older iOS Safari.

**Why it happens:** `crypto.randomUUID()` requires Safari 15.4+ (released March 2022). The project targets iPhone 17 Pro which ships with iOS 19+ — no issue. But if older device support matters, this would fail on iOS 14.

**How to avoid:** For iPhone 17 Pro target (iOS 19+), this is a non-issue. [ASSUMED: iPhone 17 Pro ships with iOS 19 or later — this is consistent with product timeline but not independently confirmed in this session]

### Pitfall 5: History Cap Fix Affects localStorage Cold-Start Behavior

**What goes wrong:** After the fix, cold start (localStorage cache load) will show all 60 entries immediately. This is correct behavior, but confirm that the Coach tab renders properly with 60 entries (no virtual scrolling performance issues).

**How to avoid:** The Coach tab already renders up to 60 entries via its history feed. The fix is safe.
