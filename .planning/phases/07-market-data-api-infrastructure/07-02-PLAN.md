## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → Supabase Auth | User submits AV/Polygon key from Account UI. Travels over HTTPS to Supabase. Stored in user_metadata (per-user, RLS-protected). |
| browser → app server | NO direct AV/Polygon key transmission to our Next.js app from the client — keys only flow client → Supabase. App server reads keys from authenticated session in route.ts (plan 07-03). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Information Disclosure | AV/Polygon API keys in Account.tsx | mitigate | Keys stored only in `user_metadata` via `supabase.auth.updateUser` (HTTPS, server-validated). UI displays masked form only (`••••XXXX`). Verification step 10 explicitly checks no request body POSTs the key to our app server. |
| T-07-A | Tampering | Malicious user submits oversized / non-string key | mitigate | `newAvKey.trim()` and `newPolygonKey.trim()` are passed directly; Supabase auth schema treats user_metadata values as JSON-serializable strings. No SQL injection vector (RLS + JSONB). Length limits not enforced client-side — relying on Supabase auth limits. Acceptable for a per-user secret. |
| T-07-B | Repudiation | User claims they did not save a key | accept | Supabase auth log records all `updateUser` calls with timestamps. Single-user app — repudiation risk is minimal. |
| T-07-C | Spoofing | XSS injects keys via crafted Account.tsx render | accept | React escapes all string render output. The masked key (`••••XXXX`) is the only key fragment ever rendered. Inputs use controlled state — no `dangerouslySetInnerHTML`. Standard React XSS protections suffice. |
</threat_model>
